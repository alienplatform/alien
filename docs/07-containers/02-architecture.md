# Containers - Architecture

How Alien and Horizon work together to provide a complete container platform.

## Division of Responsibilities

Alien Containers is built on top of **Horizon** (container orchestration) + Alien (infrastructure management).

| Component | Responsibility | Examples |
|-----------|---------------|----------|
| **Horizon** | Container orchestration | Scheduling replicas, replica autoscaling, WireGuard mesh, DNS, TCP proxies |
| **Alien** | Infrastructure management | Provisioning VMs, selecting instance types, creating load balancers, machine autoscaling, volume provisioning |

**Why this split?**

- **Horizon** is a standalone container orchestrator (like Kubernetes but simpler)
- **Alien** handles infrastructure concerns specific to BYOC deployments
- Clean separation: Horizon could be used without Alien, Alien uses Horizon as one of its execution targets

## State Model

Alien Containers use a two-layer state model:

```
StackState (Alien - Infrastructure)
   ↓
Horizon Cluster (Horizon - Containers)
```

### Layer 1: StackState (Alien)

Cloud resources and their status. Modified with deployment lock.

```rust
struct StackState {
    platform: Platform,  // AWS, GCP, Azure
    resources: HashMap<String, ResourceState>,
}

struct ResourceState {
    resource_type: ResourceType,  // ComputePool, Service
    status: String,               // "running", "pending", etc.
    outputs: HashMap<String, serde_json::Value>,
}
```

**For containers, outputs include:**
```rust
{
    "horizon_cluster_id": "workspace-123/project-456/deployment-789/cluster-001",
    "image_uri": "123.dkr.ecr.../api:v1",
    "load_balancer_dns": "api-lb.us-east-1.elb.amazonaws.com",
    "volumes": [
        { "ordinal": 0, "volumeId": "vol-abc", "zone": "us-east-1a" }
    ]
    // Note: Tokens stored in DeploymentConfig, not stack outputs
}
```

### Layer 2: Horizon Cluster State

Managed by Horizon Durable Object (no lock needed):

```rust
// In Horizon's SQLite
struct Cluster {
    machines: Vec<Machine>,    // Machine resources and status
    containers: Vec<Container>, // Container definitions
    replicas: Vec<Replica>,    // Replica assignments
}
```

## Horizon Integration

### Setup Flow

**Before Deployment (Platform Preparation):**

**1. Platform ensures Horizon clusters exist (one per ContainerCluster resource):**
```typescript
// In platform when building DeploymentConfig
// For each ContainerCluster resource in the stack
for (const [resourceId, clusterResource] of stack.containerClusters) {
  // Generate deterministic cluster ID
  const clusterId = `${deployment.workspaceId}/${deployment.projectId}/${deployment.id}/${resourceId}`;

  // Get or create cluster configuration
  let horizonConfig = deployment.horizonConfig?.clusters?.[resourceId];

  if (!horizonConfig) {
    // Create cluster in Horizon using platform JWT
    // Note: Platform analyzes containers and generates capacity groups during preflights
    const platformJwt = await generatePlatformJwt({ scopes: ["cluster:*"] });
    const response = await horizonClient.createCluster({
      clusterId,
      name: `${projectName}-${deploymentName}-${resourceId}`,
      containerCidr: "10.244.0.0/16",
      platform: deployment.platform,
      capacityGroups: clusterResource.capacityGroups,  // Generated by preflights
    }, {
      headers: { Authorization: `Bearer ${platformJwt}` }
    });

    // Tokens returned ONCE ONLY - encrypt and store in platform database
    const project = await db.selectFrom("projects")
      .select("envEncryptionKey")
      .where("id", "=", deployment.projectId)
      .executeTakeFirstOrThrow();

    // Store encrypted tokens in deployment.horizon_config (platform DB)
    const updatedHorizonConfig = deployment.horizonConfig || { clusters: {} };
    updatedHorizonConfig.clusters[resourceId] = {
      clusterId,
      managementToken: encryptEnvironmentVariableValue(response.managementToken, project.envEncryptionKey),
      machineToken: encryptEnvironmentVariableValue(response.machineToken, project.envEncryptionKey),
      createdAt: new Date().toISOString(),
    };

    await db.updateTable("deployments")
      .set({ horizonConfig: updatedHorizonConfig })
      .where("id", "=", deployment.id)
      .execute();

    horizonConfig = updatedHorizonConfig.clusters[resourceId];
  }

  // Decrypt tokens for use in deployment
  const project = await getProject(db, deployment.projectId);
  const managementToken = decryptEnvironmentVariableValue(horizonConfig.managementToken, project.envEncryptionKey);
  const machineToken = decryptEnvironmentVariableValue(horizonConfig.machineToken, project.envEncryptionKey);

  // Add to DeploymentConfig (management token for API calls)
  deploymentConfig.horizonConfig.clusters[resourceId] = {
    clusterId,
    managementToken,  // For alien-deployment API calls
  };

  // Add machine token as built-in environment variable (will be synced to vault)
  // Variable name derived from ContainerCluster resource ID
  // Example: resourceId="compute" → "ALIEN_HORIZON_MACHINE_TOKEN_COMPUTE"
  // Example: resourceId="gpu-pool" → "ALIEN_HORIZON_MACHINE_TOKEN_GPU_POOL"
  builtInEnvVars.push({
    name: `ALIEN_HORIZON_MACHINE_TOKEN_${resourceId.toUpperCase().replace(/-/g, '_')}`,
    value: machineToken,
    type: "secret",  // Will be synced to vault during Provisioning
    targetFunctions: null,
  });
}
```

**Security properties:**
- ✅ Tokens encrypted at rest in platform database
- ✅ Management token passed directly to alien-deployment (ephemeral)
- ✅ Machine token synced to cloud vault (Parameter Store/Secret Manager/Key Vault)
- ✅ Never exposed in user data or launch templates

**2. During Provisioning: Alien syncs machine token to vault:**
```rust
// Machine token was added as built-in env var by platform
// Variable name pattern: ALIEN_HORIZON_MACHINE_TOKEN_{RESOURCE_ID}
// Example for ContainerCluster with id="compute":
// { name: "ALIEN_HORIZON_MACHINE_TOKEN_COMPUTE", value: "hj_...", type: "secret" }

// sync_secrets_to_vault() automatically syncs it to cloud storage:
// - AWS: Secrets Manager: myapp-prod-secrets-ALIEN_HORIZON_MACHINE_TOKEN_COMPUTE ($0.40/mo)
// - GCP: Secret Manager: myapp-prod-secrets-ALIEN_HORIZON_MACHINE_TOKEN_COMPUTE ($0.06/mo)
// - Azure: Key Vault: myapp-prod-secrets/ALIEN_HORIZON_MACHINE_TOKEN_COMPUTE (FREE)
```

**3. Alien provisions ComputePools (ASGs) with secure token retrieval:**
```rust
// User data script fetches machine token from vault at boot (SECURE)
// Vault prefix: {resourcePrefix}-secrets (e.g., "myapp-prod-secrets")
// Token key: ALIEN_HORIZON_MACHINE_TOKEN_{containerClusterResourceId}
let vault_prefix = format!("{}-secrets", ctx.resource_prefix);
let machine_token_key = format!("ALIEN_HORIZON_MACHINE_TOKEN_{}", 
    config.id.to_uppercase().replace('-', "_"));  // config.id is the ContainerCluster resource ID

let user_data = format!(r#"#!/bin/bash
set -euo pipefail

INSTANCE_ID=$(ec2-metadata --instance-id | cut -d " " -f 2)
ZONE=$(ec2-metadata --availability-zone | cut -d " " -f 2)
REGION=$(echo $ZONE | sed 's/[a-z]$//')

# Fetch machine token from Secrets Manager (using EC2 instance IAM role)
# Secret naming: {vault_prefix}-{key}
MACHINE_TOKEN=$(aws secretsmanager get-secret-value \
  --secret-id {vault_prefix}-{machine_token_key} \
  --region $REGION \
  --query SecretString \
  --output text)

# Install containerd, WireGuard, CNI plugins
apt-get update && apt-get install -y containerd wireguard-tools awscli

# Install horizond
curl -L https://releases.horizon.dev/horizond -o /usr/local/bin/horizond
chmod +x /usr/local/bin/horizond

# Start horizond (token in memory, never written to disk)
cat > /etc/systemd/system/horizond.service <<EOF
[Unit]
Description=Horizon Machine Agent
After=network.target

[Service]
Environment="CLUSTER_ID={cluster_id}"
Environment="MACHINE_TOKEN=$MACHINE_TOKEN"
Environment="API_URL=https://horizon.alien.dev"
Environment="MACHINE_ID=$INSTANCE_ID"
Environment="CONTAINER_CIDR=10.244.${{ORDINAL}}.0/24"
ExecStart=/usr/local/bin/horizond
Restart=always

[Install]
WantedBy=multi-user.target
EOF

systemctl enable horizond
systemctl start horizond
"#, 
    cluster_id = cluster_config.cluster_id,
    vault_prefix = vault_prefix,
    machine_token_key = machine_token_key,
);
```

**Security benefits:**
- ✅ Token fetched from Secrets Manager (encrypted with KMS)
- ✅ Access controlled via IAM role (only this agent's EC2 instances)
- ✅ Never visible in launch template or cloud console
- ✅ Audit trail via CloudTrail

**5. Alien creates load balancer (if container has .expose()):**
```rust
// For services with public exposure
let load_balancer_target = if !service.expose.is_empty() {
    let load_balancer = create_load_balancer(CreateLoadBalancerRequest {
        name: format!("{}-{}-lb", stack_name, service.id),
        vpc_id: vpc_id,
        subnets: public_subnets,
        ports: service.expose.ports.clone(),
    }).await?;
    
    outputs.insert("load_balancer_dns", load_balancer.dns_name);
    
    // Build platform-specific config
    Some(match ctx.platform {
        Platform::Aws => LoadBalancerTarget::Aws {
            target_group_arn: load_balancer.target_group_arn,
        },
        Platform::Gcp => LoadBalancerTarget::Gcp {
            backend_service_url: load_balancer.backend_service_url,
        },
        Platform::Azure => LoadBalancerTarget::Azure {
            backend_pool_id: load_balancer.backend_pool_id,
            resource_group: ctx.resource_group.clone(),
        },
    })
} else {
    None
};
```

**6. Alien creates container in Horizon:**
```rust
// After ComputePool is ready, during Provisioning phase
// Look up cluster config for this container's cluster
let cluster_resource_id = &service.cluster.id;  // Container references which cluster it belongs to
let cluster_config = horizon_config.clusters.get(cluster_resource_id)?;

let container_request = CreateContainerRequest {
    name: service.id.clone(),
    capacity_group: service.capacity_group.clone(),  // Auto-assigned by preflights
    image: image_uri,
    resources: ResourceRequirements {
        cpu: ResourceSpec {
            min: service.cpu.to_string(),
            desired: service.cpu.to_string(),
        },
        memory: ResourceSpec {
            min: service.memory.clone(),
            desired: service.memory.clone(),
        },
        ephemeral_storage: service.ephemeral_storage.clone(),
        gpu: service.gpu.clone(),
    },
    autoscaling: if service.stateful {
        None
    } else {
        Some(AutoscalingConfig {
            min: service.min_replicas,
            desired: service.min_replicas,
            max: service.max_replicas,
            target_cpu_percent: 70.0,
            target_memory_percent: 80.0,
        })
    },
    replicas: if service.stateful {
        Some(service.replicas)
    } else {
        None
    },
    stateful: service.stateful,
    ports: service.ports.clone(),
    env: build_environment(&service, stack_state),
    load_balancer_target,  // Platform-specific enum
};

// Call Horizon API with management token (hm_...)
horizon_client.create_container(
    &cluster_config.cluster_id,
    container_request,
    &cluster_config.management_token
).await?;
```

**7. For stateful containers, Alien provisions volumes BEFORE calling Horizon:**

**Note:** This happens in the `ProvisionVolumes` controller state (see deployment flow below).

```rust
// In ContainerController::provision_volumes()
let mut volumes = Vec::new();
for ordinal in 0..service.replicas {
    let zone = zones[ordinal % zones.len()];  // Distribute across AZs
    
    // Create volume using platform-specific API
    let volume_target = match ctx.platform {
        Platform::Aws => {
            let volume = ec2_client.create_volume(CreateVolumeRequest {
                size: parse_storage_size(&service.persistent_storage),
                availability_zone: zone.clone(),
                volume_type: "gp3".to_string(),
                tags: vec![
                    Tag { key: "alien:service".into(), value: service.id.clone() },
                    Tag { key: "alien:ordinal".into(), value: ordinal.to_string() },
                ],
            }).await?;
            
            VolumeTarget::Aws {
                volume_id: volume.volume_id,
                zone: zone.clone(),
            }
        }
        Platform::Gcp => {
            let disk = compute_client.create_disk(CreateDiskRequest {
                name: format!("{}-{}-{}", stack_name, service.id, ordinal),
                size_gb: parse_storage_size(&service.persistent_storage),
                zone: zone.clone(),
                disk_type: "pd-ssd".to_string(),
            }).await?;
            
            VolumeTarget::Gcp {
                disk_name: disk.name,
                zone: zone.clone(),
            }
        }
        Platform::Azure => {
            let disk = compute_client.create_managed_disk(CreateDiskRequest {
                name: format!("{}-{}-{}", stack_name, service.id, ordinal),
                size_gb: parse_storage_size(&service.persistent_storage),
                location: zone.clone(),
                sku: "Premium_LRS".to_string(),
                resource_group: ctx.resource_group.clone(),
            }).await?;
            
            VolumeTarget::Azure {
                disk_id: disk.id,
                zone: zone.clone(),
                resource_group: ctx.resource_group.clone(),
            }
        }
    };
    
    volumes.push(VolumeRegistration {
        ordinal,
        volume: volume_target,
    });
}

self.volumes = volumes;  // Store for passing to Horizon in CreateHorizonContainer
```

### Runtime Flow

**Horizon scheduler (every 5 seconds):**
1. Calculate desired replicas based on metrics
2. Assign replicas to machines with capacity
3. Update assignments in SQLite

**horizond on each machine (every 5 seconds):**
1. Send heartbeat to Horizon
2. Receive cluster state (WireGuard mesh config)
3. Receive container assignments
4. Start/stop containers to match assignments
5. Report metrics back to Horizon

**Alien infrastructure actor (every 60 seconds, with lock):**
1. Query Horizon capacity plan API
2. Apply scaling recommendations to ASGs per group
3. Poll container status for deployment completion

### Public Networking

**Alien's role (during setup):**
- Create load balancers (ALB, NLB, etc.)
- Create target groups with health checks
- Store target group ARN in Horizon cluster config

**Horizon control plane's role:**
- Allocate host ports for public containers
- Include target_group_arn in replica assignments

**horizond's role (at runtime):**
- Run TCP proxies (host port → container IP)
- Register/deregister load balancer targets (machine_id:host_port)
- Uses cloud credentials (IAM role/service account) to call ELB/GCP/Azure APIs

**Why horizond manages targets:**
- Replicas scale every 5 seconds (Horizon scheduler)
- Alien infrastructure actors poll every 60 seconds
- 55-second delay would send traffic to dead replicas
- horizond acts instantly when starting/stopping containers

**Traffic flow:**
```
Internet
  → ALB (created by Alien)
  → Machine:8001 (target registered by horizond)
  → horizond TCP proxy (created by horizond)
  → Container (scheduled by Horizon)
```

## Why This Architecture Works

**Outbound-only communication:**
- Machines only make HTTPS calls (polling Horizon)
- Works in restricted environments, behind corporate firewalls
- No inbound firewall rules needed for orchestration

**Stateless control plane:**
- Horizon Durable Object stores all state in SQLite
- Can restart without losing cluster state
- Each cluster isolated in its own Durable Object

**Clear separation of concerns:**
- Alien handles infrastructure (knows about AWS/GCP/Azure APIs)
- Horizon handles orchestration (cloud-agnostic)
- Clean API boundary between them

**Simple for BYOC:**
- Customer runs: machines + containers (their workload)
- Alien runs: infrastructure controllers + Horizon control plane
- Lower operational overhead than Kubernetes

## Comparison with Kubernetes

|| Aspect | Kubernetes | Alien Containers |
||--------|-----------|------------------|
|| **Control Plane Location** | Customer account | Multi-tenant SaaS (Horizon) |
|| **State Storage** | etcd (customer-managed) | SQLite in Durable Object |
|| **Communication** | Inbound (kubelet → API server) | Outbound (horizond polls) |
|| **Infrastructure Provisioning** | Manual (separate tool) | Automatic (Alien) |
|| **Instance Selection** | Manual (user picks) | Automatic (Alien) |
|| **Customer Operates** | etcd, API server, controllers | Only machines + containers |
|| **Cost Overhead** | $200-300/month minimum | $0 (included) |
|| **Scale Limit** | 5000+ nodes | ≤10 machines per cluster |
|| **Complexity** | Very high (many components) | Low (Alien + Horizon) |

Alien Containers achieves the same end result (running containers with autoscaling, networking, service discovery) but with a fundamentally simpler architecture optimized for BYOC deployments.
