# KV

Key-Value storage that works across all platforms. The API is minimal by design—only operations that all backends support efficiently.

## Platform Mapping

| Platform | Backend |
|----------|---------|
| AWS | DynamoDB |
| GCP | Firestore |
| Azure | Table Storage |
| Kubernetes | Redis |
| Local | Redis (or file-based) |

## Design Principles

The KV API uses the lowest common denominator across all backends:

- **Standard operations only**: GET, PUT, DELETE, EXISTS, SCAN. No complex conditional writes beyond `put_if_not_exists`.
- **Prefix-based queries**: No secondary indexes. Use composite keys and prefix scans.
- **TTL as soft hint**: Items expire logically but physical deletion timing varies by backend.
- **Universal size limits**: 24KB values, 512-byte keys. Works everywhere.

## Core API

```rust
pub trait Kv: Send + Sync {
    async fn get(&self, key: &str) -> Result<Option<Vec<u8>>>;
    async fn put(&self, key: &str, value: Vec<u8>, options: Option<PutOptions>) -> Result<bool>;
    async fn delete(&self, key: &str) -> Result<()>;
    async fn exists(&self, key: &str) -> Result<bool>;
    async fn scan_prefix(&self, prefix: &str, limit: Option<usize>, cursor: Option<String>) -> Result<ScanResult>;
}

pub struct PutOptions {
    pub ttl: Option<Duration>,      // Soft expiry hint
    pub if_not_exists: bool,        // Atomic create
}
```

### Size Constraints

| Constraint | Limit | Rationale |
|------------|-------|-----------|
| Key size | 512 bytes | Safe across all backends |
| Value size | 24 KB | Azure Table Storage constraint after encoding |
| Key charset | `a-zA-Z0-9-_:.` | Avoids problematic chars (`/`, `\`, `#`, `?`) |

Validation happens at the client layer before backend calls. Invalid inputs fail fast with clear errors.

### Atomic Create

The `if_not_exists` option maps to backend primitives:

| Backend | Primitive |
|---------|-----------|
| DynamoDB | `PutItem` with `attribute_not_exists` condition |
| Firestore | `create()` with `Precondition::DoesNotExist` |
| Table Storage | `InsertEntity` (returns 409 on conflict) |
| Redis | `SETNX` |

Returns `true` if created, `false` if already exists.

## TTL Behavior

TTL is a **soft hint**. Expired items appear absent on reads, but physical deletion timing varies.

### Logical Expiry (All Backends)

All operations check expiry client-side:

```rust
async fn get(&self, key: &str) -> Result<Option<Vec<u8>>> {
    let item = self.backend_get(key).await?;
    if let Some(expires_at) = item.expires_at {
        if Utc::now() >= expires_at {
            return Ok(None);  // Logically expired
        }
    }
    Ok(item.value)
}
```

### Physical Deletion (Backend-Specific)

| Backend | Physical Deletion |
|---------|-------------------|
| DynamoDB | Eventual (up to 48 hours) |
| Firestore | Eventual (hours) |
| Table Storage | Manual cleanup required |
| Redis | Immediate on access, lazy otherwise |

Don't rely on TTL for guaranteed physical deletion. Use explicit delete for cleanup-critical paths.

## Scan Contract

`scan_prefix` returns an unordered subset of keys matching the prefix:

- **No ordering guarantees**: Results come in backend-natural order
- **May return duplicates**: Clients must de-duplicate across pages
- **May return fewer than limit**: Not guaranteed to fill even if more data exists
- **Cursors can expire**: No TTL guarantees on cursor validity

```rust
let mut all_items = HashSet::new();
let mut cursor = None;

loop {
    let result = kv.scan_prefix("user:", Some(100), cursor).await?;
    for (key, value) in result.items {
        all_items.insert(key);  // De-duplicate
    }
    cursor = result.next_cursor;
    if cursor.is_none() { break; }
}
```

## Backend Notes

### DynamoDB

Uses a hash-bucketed schema to distribute load:

```
Table: alien-kv
Primary Key: pk (hash bucket), sk (full key)
Attributes: value (Binary), ttl (Number)
```

Prefix scans within a partition use `begins_with(sk, :prefix)`. Cross-partition scans require fan-out.

### Firestore

Uses document IDs for keys, with prefix range queries:

```rust
collection("kv")
    .order_by(FieldPath::document_id())
    .start_at(prefix)
    .end_at(format!("{}\u{f8ff}", prefix))
```

Native TTL via Firestore's TTL policy on `expires_at` field. Disable single-field index on TTL to avoid hotspots.

### Azure Table Storage

No native TTL—uses client-side filtering plus manual cleanup jobs.

Hash-based partitioning for load distribution:

```rust
fn split_key(&self, key: &str) -> (String, String) {
    let partition = format!("p{}", self.hash_bucket(key));
    (partition, key.to_string())
}
```

Prefix scans require fan-out across all partitions.

### Redis

Most capable backend—native `SETNX`, `SCAN MATCH`, immediate TTL:

```rust
// Atomic create with TTL
redis::cmd("SET").arg(key).arg(&value).arg("NX").arg("EX").arg(ttl_secs)
```

`SCAN` returns unordered results and may duplicate keys across pages.

## Performance Characteristics

| Backend | Latency | Throughput | Scalability | Consistency |
|---------|---------|------------|-------------|-------------|
| DynamoDB | Low | Very High | Excellent | Configurable* |
| Firestore | Medium | High | Good | Strong |
| Table Storage | Low | High | Good | Strong** |
| Redis | Very Low | Very High | Good*** | Strong |

*DynamoDB reads can be eventual or strong (strong is optional per read)  
**Table Storage provides strong read-after-write consistency in primary region  
***Redis scalability limited by memory; requires clustering for horizontal scale

## Hot Partition Considerations

**DynamoDB**: If using centralized partition key, consider hash bucketing to avoid hot partitions.

**Table Storage**: Evenly distribute PartitionKey values to avoid throttling. Cross-partition queries and scans are slower and page more frequently.

## Usage Patterns

### Composite Keys

Use structured keys instead of secondary indexes:

```
user:{user_id}:profile          → user profile data
user:{user_id}:session:{id}     → user sessions
order:{date}:{order_id}         → orders by date
```

Scan by prefix to query related data:

```rust
// Get all sessions for user
let sessions = kv.scan_prefix("user:123:session:", None, None).await?;
```

### Leasing

Use `put_if_not_exists` with TTL for distributed locks:

```rust
let lease_key = format!("lock:{}", resource_id);
let success = kv.put(&lease_key, lease_data, Some(PutOptions {
    ttl: Some(Duration::from_secs(30)),
    if_not_exists: true,
})).await?;

if success {
    // Acquired lease
} else {
    // Already held by another process
}
```

### Separate Keys for State

Instead of updating fields within a value, use separate keys:

```
req:{id}           → immutable request data
req:{id}:state     → current state (overwritten)
req:{id}:lease     → lease info (TTL-based)
```

This avoids read-modify-write races and works with the simple PUT/GET API.

## Case Study: Command Server

The command server demonstrates why this minimal API is sufficient for complex applications requiring state management, leasing, and expiry handling.

### Request Lifecycle

Command requests flow through states: `PENDING_UPLOAD` → `PENDING` → `DISPATCHED` → `SUCCEEDED/FAILED/EXPIRED`.

```rust
// Atomic create to prevent duplicates
async fn create_request(kv: &dyn Kv, request: &CommandRequest) -> Result<()> {
    let key = format!("req:{}", request.id);
    let value = serde_json::to_vec(request)?;
    
    let success = kv.put(&key, value, Some(PutOptions {
        ttl: None,
        if_not_exists: true,
    })).await?;
    
    if !success {
        return Err(Error::AlreadyExists);
    }
    Ok(())
}

// State transition using separate key (eventual consistency is acceptable)
async fn transition_state(kv: &dyn Kv, request_id: &str, to_state: RequestState) -> Result<()> {
    let state_key = format!("req:{}:state", request_id);
    let value = serde_json::to_vec(&StateInfo { state: to_state, updated_at: Utc::now() })?;
    kv.put(&state_key, value, None).await?;
    Ok(())
}
```

### Leasing System

Kubernetes deployments poll for work by acquiring leases on PENDING requests:

```rust
// Key schema:
// target:{target_ref}:pending:{created_timestamp}:{request_id} → minimal data
// req:{request_id}:lease → lease info (with TTL)

async fn acquire_lease(kv: &dyn Kv, target_ref: &str, lease_duration: Duration) -> Result<Option<LeaseInfo>> {
    // Find oldest pending request for this target
    let prefix = format!("target:{}:pending:", target_ref);
    let scan_result = kv.scan_prefix(&prefix, Some(10), None).await?;
    
    for (index_key, _) in scan_result.items {
        let request_id = extract_request_id(&index_key);
        let lease_key = format!("req:{}:lease", request_id);
        
        // Atomic lease creation with TTL
        let lease_info = LeaseInfo {
            lease_id: generate_lease_id(),
            acquired_at: Utc::now(),
            expires_at: Utc::now() + lease_duration,
        };
        
        let success = kv.put(&lease_key, serde_json::to_vec(&lease_info)?, Some(PutOptions {
            ttl: Some(lease_duration),
            if_not_exists: true,
        })).await?;
        
        if success {
            kv.delete(&index_key).await?;  // Remove from pending index
            transition_state(kv, &request_id, RequestState::Dispatched).await?;
            return Ok(Some(lease_info));
        }
    }
    
    Ok(None)  // No available requests
}
```

### Why This Works

1. **Atomic creation** via `put_if_not_exists` prevents duplicate requests and double-leasing
2. **Efficient queries** via prefix scans with composite keys (no secondary indexes needed)
3. **Automatic cleanup** via TTL (leases auto-expire)
4. **Eventual consistency** is acceptable where the command protocol handles it
