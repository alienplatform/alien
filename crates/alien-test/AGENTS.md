# alien-test

E2E test infrastructure crate for Alien.

## Purpose

Provides reusable building blocks for integration and E2E tests that need a running alien-manager instance. Tests spin up an in-process manager with an ephemeral SQLite database and a bootstrapped admin token, interact with it through the alien-manager-api SDK client, and tear everything down automatically.

## Key types

- **`TestManager`** -- Starts a standalone alien-manager on a random port with a temp SQLite dir. Provides an authenticated SDK client and admin token. Accepts optional `TestConfig` and platform list to inject cloud credentials. Shuts down on drop.
- **`TestDeployment`** -- Wraps a deployment created via the manager. Provides `deploy()` (static constructor), `wait_until_running()`, `invoke_command()`, `upgrade()`, `check_health()`, and `destroy()`.
- **`TestAlienAgent`** -- Starts an alien-agent Docker container or Helm release for pull-model testing. Provides `start_container()`, `helm_install()`, `stop()`, and `helm_uninstall()`.
- **`TestConfig`** -- Loads cloud credentials and test resources from `.env.test` (generated by `scripts/gen-env-test.sh`). Reports which platforms have both management and target credentials available.
- **`cleanup_deployments`** -- Best-effort teardown of all deployments on a manager.
- **`cleanup_agent_containers`** -- Docker cleanup of labelled alien-agent containers.
- **`cleanup_helm_release`** -- Helm uninstall of a test release.
- **`setup_target`** -- Stub for target account provisioning (will invoke alien-deploy-cli).

## Usage

```rust
use alien_test::{TestManager, TestConfig};

#[tokio::test]
async fn example() {
    let manager = TestManager::start().await.unwrap();
    let client = manager.client();
    // ... create releases, deployments, invoke commands ...
    manager.stop().await;
}

#[tokio::test]
async fn example_with_cloud() {
    let config = TestConfig::from_env();
    let platforms = config.available_platforms();
    let manager = TestManager::start_with_config(&config, &platforms)
        .await
        .unwrap();
    // ... deploy to real cloud targets ...
    manager.stop().await;
}
```

## Terminology

- Use **"deployment"** not "agent" for deployed application instances
- Use **"manager"** not "server" for the control plane
- Use **"commands"** not "ARC" for the remote command system

## Debugging E2E Failures

**Root-cause first.** Never guess at fixes. Prove the issue before changing code.

1. **Read cloud logs** — `gcloud logging read`, CloudWatch Logs, or Azure Monitor. Always start here.
2. **Reproduce locally first** — if a binding fails on Cloud Run, test it locally with `alien dev` to isolate cloud-specific vs logic bugs.
3. **Add debug logs** — if logs are insufficient, add targeted `tracing::info!` / `tracing::error!` to the runtime or binding code, rebuild, and redeploy.
4. **Deploy minimal repros** — for Cloud Run/Lambda/Container Apps issues, deploy a minimal app (just health + the failing binding) to iterate faster than full E2E.
5. **Don't increase memory/timeouts blindly** — these mask bugs. If you suspect OOM, prove it with metrics or logs (`textPayload=~"OOM|killed"`).
6. **Check the forwarding chain** — Cloud Run requests: Cloud Run → Alien runtime (port 8080) → Bun subprocess (random port) → gRPC bindings → cloud APIs. Identify which hop fails.

## Ngrok Tunnel

Cloud-deployed functions need to reach the local manager for commands polling. The ngrok tunnel (`NGROK_AUTHTOKEN` + `NGROK_DOMAIN` from `.env.test`) forwards `https://{domain}/v1/...` to `http://localhost:{port}/v1/...`.

The `NgrokTunnel` struct is stored in `TestManager` and holds both the `Forwarder` and `Session`. Both must stay alive for the tunnel to work — dropping either closes the tunnel.
